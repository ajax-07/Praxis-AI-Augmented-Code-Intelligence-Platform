2 · The spine — one analysis, end to end

Every stage runs for real now, including the LLM call — the amber SUMMARIZING arrow marks the one config-dependent hop: which engine answers (offline stub by default, Ollama / OpenAI / Gemini via CORTEX_PROVIDER). If the provider is down or has no key, the batch stops and the analysis completes on static findings alone. Scroll horizontally if needed.

3 · Identity flows

Stateless JWT — the token is self-contained, so no user lookup happens per request. All three flows are live.

Register LIVE

  1. POST /api/v1/auth/register {email, password ≥ 8, tenantName}
  2. email exists → 409 EMAIL_ALREADY_REGISTERED
  3. else create Tenant + first User as ADMIN (BCrypt) — one transaction
  4. issue JWT {sub, tenantId, role}, HS256, 60 min → {token, "Bearer"}

Login LIVE

  1. POST /api/v1/auth/login {email, password}
  2. unknown email and wrong password → identical 401 INVALID_CREDENTIALS
  3. BCrypt match → fresh JWT

Every request LIVE

  1. Authorization: Bearer … → JwtAuthFilter parses & verifies
  2. SSE exception: no header possible → ?access_token= accepted, but only for URIs ending /events
  3. valid → PraxisPrincipal in SecurityContext; invalid → context cleared → 401
  4. modules read tenant via IdentityFacade.currentTenantId() — never from the request body

4 · Analysis state machine

QUEUED→ FETCHING→ PARSING→ ANALYZING→ SUMMARIZING→ SCORING→ COMPLETE any stage ⇢ FAILED Guards live in the Analysis aggregate: advanceTo() / markComplete() / markFailed() are no-ops once terminal — redelivered Redis jobs can never move a finished analysis backwards. Failed analyses are terminal; there is no auto-retry (XAUTOCLAIM reclaim is Phase-2 hardening). SUMMARIZING is amber because its engine is config-selected (stub by default) — and it degrades gracefully: an unreachable provider never fails the analysis, it just completes static-only.

5 · Build order from here

Phase 1 is done — pipeline, LLM providers, and console all live and smoke-tested end to end. What remains, in the order that unlocks the most:

01cache · resilience Redis response cache keyed sha256(src + promptVersion + model) (set cache_hit=true on hits — start actually reading promptVersion), Resilience4j retry + circuit breaker around the client (graceful degradation already exists), and a price book mapping (provider, model, tokens) → cost_cents.
02recall · ledger Phase 2: pgvector embeddings + chat Q&A grounded in code units; token metering + BudgetGuard taking over the llm_call table from Cortex. Tables already exist in V1.
03hardening Env-only JWT secret & DB password, ddl-auto: validate, CORS for the console, user invites, XAUTOCLAIM reclaim for crashed workers, Markdown/PDF export, code_residency = LOCAL_ONLY forcing Ollama-only routing.
